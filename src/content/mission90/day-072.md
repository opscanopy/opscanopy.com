## The Kubernetes debugging loop: describe, logs, events

When a Pod misbehaves you don't guess and you don't restart blindly — you run the same short loop every time, from the outside in, and stop the moment the cause is obvious. Master this loop and you can debug any workload on any cluster.

**Start wide: `kubectl get pods`.** The `STATUS` and `RESTARTS` columns are your first clue. `Running` with climbing restarts means the container keeps dying. `ImagePullBackOff`, `CrashLoopBackOff`, `Pending` and `CreateContainerConfigError` each name a *different* failure, and reading them on sight is half the battle.

**Then narrow: `kubectl describe pod <name>`.** Scroll straight to the **Events** section at the bottom — the single most useful output in Kubernetes. The control plane records every decision it made about your Pod (scheduled here, pulling that image, failed because…) as timestamped events, and `describe` prints them per object.

**Then go inside: `kubectl logs <name>`.** These are your *application's own words* — whatever it wrote to stdout and stderr. If the container has already crashed and restarted, add `-p` (`--previous`) to read the dead instance's final output; the running container may be empty or mid-boot, but the previous one printed exactly why it died.

**Widen when needed: `kubectl get events --sort-by=.metadata.creationTimestamp`** is the cluster-wide timeline across every object, and `kubectl exec -it <name> -- sh` drops you inside a *running* container to check config and DNS live.

The four states you'll meet most:

- **ImagePullBackOff** — the kubelet can't pull the image (wrong tag, private registry, typo). Events show `ErrImagePull` first, then the backoff.
- **CrashLoopBackOff** — the container starts, exits and restarts on a growing back-off. The image is fine; the *process* dies. `logs -p` tells you why.
- **Pending** — the scheduler can't place the Pod on any node (insufficient CPU/memory, an unbound volume, a taint). Nothing has even started.
- **CreateContainerConfigError** — a referenced ConfigMap or Secret key doesn't exist, so the kubelet can't build the container's config. Events name the missing object.

> **Real world:** A Pod that won't start is a car that won't start. `kubectl get pods` is the dashboard warning light — it tells you *something* is wrong (CrashLoopBackOff) but not why. `kubectl describe` is the mechanic plugging in the diagnostic reader and printing the fault history — the Events. And `kubectl logs` is popping the hood to hear the engine itself. You read them in that order and stop the moment the cause is obvious.

The split that makes this fast: **logs are the app talking; events are Kubernetes talking.** A `Connection refused` in the logs is the app's problem; a `Failed to pull image` in the Events is the platform's. This is exactly the data that tools like **k9s** — the open-source terminal UI over kubectl that SREs leave running all day — surface continuously: the same Events and Pod status, just refreshed for you. Learn to pull it by hand first, and the tools become a convenience rather than a crutch.

## Hands-On Lab

Budget about 30 minutes. This runs entirely locally on the `kind` cluster you created on Day 67 — Kubernetes in Docker on WSL2, no cloud and no bill. You'll deploy a healthy app, then break it three ways and diagnose each from `describe`, `logs` and `events` alone. Pod name suffixes, ReplicaSet hashes, ages, IPs and node names are unique to each run — yours will differ from the samples.

```bash
# 1. Confirm your kind cluster is up (from Day 67). One Ready node is enough.
kubectl get nodes
# Output (AGE and VERSION will differ):
# NAME                 STATUS   ROLES           AGE   VERSION
# kind-control-plane   Ready    control-plane   3d    v1.31.0
```

```yaml
# 2. deploy.yaml — a healthy linkstash Deployment we'll break three ways.
apiVersion: apps/v1
kind: Deployment
metadata:
  name: linkstash
spec:
  replicas: 1
  strategy: { type: Recreate }
  selector:
    matchLabels: { app: linkstash }
  template:
    metadata:
      labels: { app: linkstash }
    spec:
      containers:
        - name: web
          image: nginx:1.27
          ports:
            - containerPort: 80
```

```bash
# 3. Apply it and confirm the Pod reaches Running before you break anything.
kubectl apply -f deploy.yaml
kubectl get pods
# Output (Pod suffix and AGE will differ):
# deployment.apps/linkstash created
# NAME                        READY   STATUS    RESTARTS   AGE
# linkstash-7c9f8b6d4-abcde   1/1     Running   0          15s
```

```bash
# 4. BREAK #1 — point at a tag that does not exist. Watch STATUS, not just READY.
kubectl set image deploy/linkstash web=nginx:9.9.9
kubectl get pods
# Output (new ReplicaSet hash; STATUS is the tell):
# deployment.apps/linkstash image updated
# NAME                        READY   STATUS             RESTARTS   AGE
# linkstash-6d4b9c8f7-fghij   0/1     ImagePullBackOff   0          40s
```

```bash
# 5. Diagnose #1 — describe and read the Events at the bottom. This IS the answer.
kubectl describe pod -l app=linkstash | sed -n '/Events:/,$p'
# Output (trimmed to the Events block; timestamps and node differ):
# Events:
#   Type     Reason     Age                From               Message
#   ----     ------     ----               ----               -------
#   Normal   Pulling    26s (x2 over 40s)  kubelet            Pulling image "nginx:9.9.9"
#   Warning  Failed     25s (x2 over 39s)  kubelet            Failed to pull image "nginx:9.9.9": ... manifest unknown
#   Warning  Failed     25s (x2 over 39s)  kubelet            Error: ErrImagePull
#   Normal   BackOff    12s (x3 over 39s)  kubelet            Back-off pulling image "nginx:9.9.9"
#   Warning  Failed     12s (x3 over 39s)  kubelet            Error: ImagePullBackOff
```

```bash
# 6. BREAK #2 — restore the good image but make the container crash on boot.
kubectl patch deploy/linkstash --type=json \
  -p='[{"op":"add","path":"/spec/template/spec/containers/0/command","value":["sh","-c","echo FATAL: config missing; exit 1"]}]'
kubectl set image deploy/linkstash web=nginx:1.27
kubectl get pods
# Output (RESTARTS climbs — the process starts and dies on a loop):
# deployment.apps/linkstash patched
# deployment.apps/linkstash image updated
# NAME                        READY   STATUS             RESTARTS      AGE
# linkstash-58c7d9f6b-klmno   0/1     CrashLoopBackOff   3 (25s ago)   90s
```

```bash
# 7. Diagnose #2 — the current container is dead, so read the PREVIOUS one's logs.
kubectl logs --previous deploy/linkstash
# Output (the app's final words before it exited):
# FATAL: config missing
```

```yaml
# 8. deploy-secret.yaml — restore a clean spec but pull env from a Secret that
#    does not exist. Re-applying replaces the crashing command from step 6.
apiVersion: apps/v1
kind: Deployment
metadata:
  name: linkstash
spec:
  replicas: 1
  strategy: { type: Recreate }
  selector:
    matchLabels: { app: linkstash }
  template:
    metadata:
      labels: { app: linkstash }
    spec:
      containers:
        - name: web
          image: nginx:1.27
          envFrom:
            - secretRef:
                name: linkstash-secrets   # this Secret was never created
          ports:
            - containerPort: 80
```

```bash
# 9. BREAK #3 — apply it. A missing config source is a different state again.
kubectl apply -f deploy-secret.yaml
kubectl get pods
# Output (not ImagePull, not CrashLoop — the kubelet can't even build the config):
# deployment.apps/linkstash configured
# NAME                        READY   STATUS                       RESTARTS   AGE
# linkstash-9f6c5b8d7-pqrst   0/1     CreateContainerConfigError   0          20s
```

```bash
# 10. Diagnose #3 — the cluster-wide timeline names the missing object directly.
kubectl get events --sort-by=.metadata.creationTimestamp | tail -3
# Output (the last line is the whole diagnosis):
# LAST SEEN   TYPE      REASON    OBJECT                          MESSAGE
# 18s         Normal    Pulled    pod/linkstash-9f6c5b8d7-pqrst   Container image "nginx:1.27" already present on machine
# 2s          Warning   Failed    pod/linkstash-9f6c5b8d7-pqrst   Error: secret "linkstash-secrets" not found
```

```bash
# 11. Clean up — delete the Deployment so the next day starts from a clean cluster.
kubectl delete -f deploy-secret.yaml
# Output:
# deployment.apps "linkstash" deleted
```

Read the three failures back to back: the image failure never *ran* (Events, not logs), the crash *ran and died* (logs -p), and the config failure never *built* (Events name the missing Secret). Same loop, three states — that is the toolkit the Kubernetes Chaos mission will test tomorrow.

## Common Errors & Fixes

These three trip people up the moment they debug Kubernetes for real. Read the error text slowly — parsing it is the actual skill.

> **Common error:** Asking `kubectl` for a resource type that doesn't exist, usually a typo or a plural slip:
>
> ```text
> error: the server doesn't have a resource type "deploment"
> ```
>
> **Why:** `kubectl` maps the noun you typed against the API server's known resource types (and their short names) and found nothing called `deploment`. Because it's asking the *server*, the same message appears when a Custom Resource Definition isn't installed on this cluster — the type genuinely isn't there.
>
> **Fix:** Run `kubectl api-resources` to list every valid type with its short names, then use the real one — `deployments` or `deploy`. For a CRD, install the operator/CRD first, then retry.
>
> **How you'd spot it in prod:** A script that worked yesterday failing with this after a cluster upgrade often means an API version was removed (a beta API graduated and the old one was dropped), not a typo — check the release notes and `kubectl api-resources`.

> **Common error:** A Pod sits in `Pending` forever and the Events show a scheduling failure:
>
> ```text
> Warning  FailedScheduling  default-scheduler  0/1 nodes are available: 1 Insufficient memory. preemption: 0/1 nodes are available: 1 No preemption victims found.
> ```
>
> **Why:** The scheduler couldn't find a node that satisfies the Pod's resource *requests* (or a `nodeSelector`/taint excludes every node). On a single-node kind cluster this is almost always a `requests.memory` or `requests.cpu` larger than the one node can offer, so nothing ever starts.
>
> **Fix:** Compare the Pod's requests against `kubectl describe node` (the Allocatable vs Allocated block). Lower the requests, or fix the `nodeSelector`/toleration that's fencing the Pod off.
>
> **How you'd spot it in prod:** A wave of `Pending` pods right after a deploy usually means a requests bump outgrew the cluster — check node capacity and whether the Cluster Autoscaler is adding nodes.

> **Common error:** Reaching for `logs -p` on a Pod that has never restarted:
>
> ```text
> Error from server (BadRequest): previous terminated container "web" in pod "linkstash-7c9f8b6d4-abcde" not found
> ```
>
> **Why:** `-p` / `--previous` asks for the container's *prior* terminated instance, but this container is on its first run — `RESTARTS` is 0, so there is no previous instance to show.
>
> **Fix:** Drop `-p` to read the current container's logs. Only add `--previous` once `kubectl get pods` shows `RESTARTS` above 0.
>
> **How you'd spot it in prod:** Muscle-memory `logs -p` on a slow-but-healthy Pod returns this BadRequest — check the `RESTARTS` column first; if it's 0, the current logs are the ones you want.

## Go Deeper

Optional extras if you have ~30 more minutes:

- **5 min** — On your kind cluster run `kubectl describe node kind-control-plane` and find the Allocatable vs Allocated resources block — the exact numbers the scheduler checks when a Pod goes Pending.
- **10 min** — Install [k9s](https://k9scli.io/) and watch the Events and Pod status update live as you re-break the Deployment — the same `describe`/`logs` data you pulled by hand, refreshed continuously.
- **15 min** — Read `kubectl debug --help`, then attach an ephemeral debug container to a running Pod: `kubectl debug -it <pod> --image=busybox --target=web`. It's the modern (1.31) way to get a shell alongside a crashing or distroless container that has no shell of its own.
